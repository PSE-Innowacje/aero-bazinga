# Phase 03 Plan 02: Operations Workflow

## Objective
Implement status transitions (supervisor confirm/reject, planner cancel), history log completeness, and comments API. Also ensure the OPS-09 auto-transition hooks are in place for Phase 4 flight order integration.

## Tasks

### Task 1: Add status transition endpoint
type: auto

Add POST /api/operations/:id/status to operations.ts:

Allowed transitions:
- 1 → 2: Supervisor only (reject) — no prerequisites
- 1 → 3: Supervisor only (confirm) — requires planned_earliest_date AND planned_latest_date to be set
- 1 → 7: Planner (their own operation) or Supervisor (cancel)
- 3 → 7: Planner (their own operation) or Supervisor (cancel)
- 4 → 7: Planner (their own operation) or Supervisor (cancel)

Request body: { to_status: number, reason?: string }

On each transition:
- Validate the operation exists
- Validate user has correct role for the transition
- For confirm (→3): validate planned dates are set
- Update status in planned_operations
- Record history entry: field_name="status", old_value=old status label, new_value=new status label
- If reason provided, add it as a history note: field_name="status_reason", old_value=null, new_value=reason
- Return updated operation

### Task 2: Add comments endpoint
type: auto

Add to operations.ts:

GET /api/operations/:id/comments:
- Returns all comments for the operation ordered by created_at ASC
- Joins user email + first_name + last_name for author display
- Requires planowanie_operacji READ permission

POST /api/operations/:id/comments:
- Accepts { comment_text: string } (max 500 chars)
- Stores comment with user_id from session, created_at = NOW()
- Requires planowanie_operacji READ permission (all roles with access can comment)
- Returns created comment with author info
