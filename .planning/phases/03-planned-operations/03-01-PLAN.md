### Task 3: Create KML parsing utility
type: auto

Create server/src/utils/kml.ts:
- parseKml(xmlString: string): { points: Array<{lat: number, lng: number}>, error?: string }
- Uses fast-xml-parser to parse KML XML
- Extracts coordinates from Placemark/LineString/coordinates or Point elements
- Validates: max 5000 points, all coordinates within Poland bounding box (lat 49.0–54.9, lng 14.1–24.2)
- Returns points array or error message
- Also validates the file has at least 2 points (need at least 2 for a route)

### Task 4: Create Haversine distance utility
type: auto

Create server/src/utils/haversine.ts:
- haversineDistance(lat1: number, lng1: number, lat2: number, lng2: number): number (returns km)
- totalRouteDistance(points: Array<{lat: number, lng: number}>): number
- Uses the standard Haversine formula (Earth radius = 6371 km)

### Task 5: Create operations router (GET list, GET by id, POST create)
type: auto

Create server/src/routes/operations.ts with:

GET /api/operations — list all operations
- Join operation_types, contact_persons, created_by user email
- Filter by status query param (optional); if status=3 default not applied server-side (list endpoint returns all; default filter is frontend concern)
- Sort by planned_earliest_date ASC NULLS LAST, then created_at ASC
- Returns array of operations with joined data

GET /api/operations/:id — single operation detail
- Full operation with joined types, contacts, creator email
- Also include operation_types and contact_persons arrays

POST /api/operations — create new operation (planner or supervisor only)
- Accepts multipart/form-data (kml file + JSON fields)
- Requires: project_reference, short_description, operation_type_ids (array), kml file
- Optional: proposed_earliest_date, proposed_latest_date, additional_info, contact_emails
- Auto-assigns operation_number from sequence; formats as YYYY-NNNN (year from NOW())
- Parses KML, validates points (500 max for storage efficiency), calculates distance
- Stores kml file to server/uploads/kml/ directory
- Inserts operation_history entry for creation event (field_name="created", old_value=null, new_value="operation created")
- Sets status=1, created_by_user_id from session

PUT /api/operations/:id — update operation
- Requires auth + planowanie_operacji READ permission (both planner and supervisor can call)
- Role-based field restrictions enforced server-side:
  - Planner (role=planner): cannot update planned_earliest_date, planned_latest_date, status, route_distance_km, kml_file_path, post_completion_notes
  - Supervisor (role=supervisor): full field access including planned dates and post_completion_notes
- Can update: project_reference, short_description, proposed_earliest_date, proposed_latest_date, additional_info, operation_type_ids, contact_emails, kml file (re-upload)
- If kml file re-uploaded: re-parse, recalculate distance
- Records every changed field to operation_history (compare old vs new values)
- Planner can only edit operations in statuses 1, 2, 3, 4, 5
